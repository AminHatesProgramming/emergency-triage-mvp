# پیش‌نویس گزارش نهایی پروژه

## عنوان

سیستم هوشمند پشتیبان تصمیم‌گیری تریاژ اورژانس

## خلاصه مدیریتی

این پروژه با هدف طراحی و پیاده‌سازی یک MVP مبتنی بر هوش مصنوعی برای کمک به تریاژ اورژانس انجام شده است. سیستم با دریافت اطلاعات قابل دسترسی در لحظه تریاژ، مانند علائم حیاتی، سن، روش ورود، شکایت اصلی و سابقه بالینی قابل پرسش، احتمال بحرانی بودن بیمار را تخمین می‌زند و یک خروجی decision-support ارائه می‌کند.

اهمیت پروژه انسانی و اجتماعی است: در شرایط ازدحام، بحران، حادثه جمعی، جنگ، کمبود منابع یا فشار کاری، تشخیص سریع‌تر بیماران پرخطر می‌تواند به کاهش رنج مردم و استفاده بهتر از ظرفیت درمانی کشور کمک کند. پروژه ادعای جایگزینی متخصص ندارد؛ هدف آن کمک به تصمیم بهتر، سریع‌تر و شفاف‌تر است.

## انطباق با درس مدیریت پروژه فناوری اطلاعات

پروژه با رویکرد Agile/Scrum سبک مدیریت شده و خروجی‌های مدیریتی زیر برای آن تولید شده است:

- backlog و sprint planning
- risk register
- status report
- model card
- architecture document
- AI usage report
- lessons learned
- محتوای پوستر و طرح ارائه

موضوع پروژه با محورهای AI-enabled system، مدیریت بحران، مستندسازی قابل ردیابی، MVP، API-first و mobile-first همخوان است.

## تغییر Scope نسبت به ثبت اولیه

در فایل ثبت اولیه کلاس، موضوع گروه به صورت «پلتفرم هوشمند ایجاد ارتباطات اجتماعی از طریق بازی‌های گروهی» دیده می‌شود. در ادامه مسیر، تیم با توجه به تاکید درس بر بحران، ارزش اجتماعی، اثر ملی و امکان ساخت MVP قابل دفاع، scope پروژه را به «سیستم هوشمند پشتیبان تصمیم‌گیری تریاژ اورژانس» تغییر داد.

دلایل این pivot:

- اثر انسانی مستقیم‌تر در شرایط بحران و ازدحام
- امکان تعریف KPIهای کمی روشن مثل AUC، Recall و FPR
- امکان ساخت demo واقعی‌تر با API و UI
- تناسب بهتر با محور مسئولیت اجتماعی و تاب‌آوری کشور
- قابلیت مستندسازی فنی و مدیریتی قوی‌تر

این تغییر scope به صورت صادقانه در گزارش ثبت شده و به عنوان یک تصمیم مدیریتی برای افزایش ارزش پروژه ارائه می‌شود.

## محصول ساخته‌شده

خروجی فعلی شامل سه بخش اصلی است:

1. مدل ML نسخه `v6`
2. backend با FastAPI
3. frontend فارسی mobile-first برای demo

API با endpointهای `/health`، `/model-info` و `/predict` آماده است. UI امکان ثبت علائم حیاتی، شکایت اصلی، سابقه مراجعه/بستری/جراحی و سابقه‌های بالینی را دارد. اگر همه داده‌ها موجود نباشند، سیستم همچنان پیش‌بینی می‌دهد و سطح کامل بودن داده را نمایش می‌دهد.

## تقویت مدیریت پروژه پس از بازخورد TA

پس از دریافت بازخورد TA، یک Sprint تکمیلی با هدف تبدیل پروژه از «خروجی فنی» به «پروژه قابل دفاع مدیریت فناوری اطلاعات» تعریف شد. خروجی این Sprint شامل موارد زیر است:

- تعریف `Agile Delivery Evidence` برای نگاشت نسخه‌های مدل و سامانه به Sprint و Deliverable
- آماده‌سازی `github-issues-seed.csv` برای ساخت GitHub Issues/Projects به عنوان ابزار مدیریت کار
- ایجاد Knowledge Base شامل sprint notes، meeting notes، technical decisions، stakeholder feedback و team playbook
- افزودن فرم بازخورد به MVP برای جمع‌آوری feedback واقعی از ۵ تا ۱۰ کاربر یا ذی‌نفع
- شفاف‌سازی نقش اعضای تیم به شکلی که هر عضو بتواند بخش ساده و واقعی خود را در ارائه توضیح دهد

این اقدام‌ها نشان می‌دهد پروژه فقط توسعه مدل AI نیست؛ بلکه شامل backlog، sprint، task owner، time tracking، knowledge management، feedback loop و مسیر operationalization است.

## رویکرد داده و جلوگیری از leakage

مدل فقط از داده‌هایی استفاده می‌کند که در لحظه تریاژ قابل دفاع هستند:

- علائم حیاتی اولیه
- شکایت اصلی بیمار
- سن، جنسیت و روش ورود
- سوابق قابل مشاهده در پرونده یا قابل پرسش از بیمار
- فیچرهای مشتق‌شده بالینی
- missingness features

داده‌های بعد از تریاژ مانند آزمایش‌ها، داروها، تصویربرداری، disposition و تشخیص‌های بعدی حذف شده‌اند. همچنین race، ethnicity و insurance تا قبل از تحلیل fairness وارد مدل نشده‌اند.

## بهبودهای فنی v6

- اصلاح واحد دما از Fahrenheit به Celsius
- اضافه شدن سابقه‌های بالینی EHR-safe
- اضافه شدن `n_surgeries`
- انتخاب featureهای سابقه‌ای با prevalence کنترل‌شده برای کاهش پیچیدگی
- پشتیبانی بهتر از ورودی ناقص با missingness features
- انتخاب threshold بر اساس validation و حالت safety-first

## متریک‌های مدل v6

حالت عملیاتی انتخاب‌شده: safety-first

| معیار | مقدار تست |
|---|---:|
| AUC | 0.8947 |
| Average Precision | 0.8034 |
| Recall | 0.9241 |
| Precision | 0.5269 |
| FPR | 0.3598 |
| Threshold | 0.2962 |

تفسیر مدیریتی: هدف اصلی پروژه از دست ندادن بیمار بحرانی است، بنابراین Recall نسبت به Precision اولویت دارد. با این حال، یک operating mode جایگزین نیز ثبت شده است که FPR را تا `0.3483` کاهش می‌دهد و برای سناریوهایی با منابع محدودتر قابل بررسی است.

## قابلیت کار با داده ناقص

در محیط واقعی همیشه همه اطلاعات کامل نیست. بنابراین API و frontend طوری طراحی شده‌اند که با چند آیتم نیز کار کنند. خروجی سامانه علاوه بر probability، موارد زیر را نمایش می‌دهد:

- `data_completeness`
- `confidence_band`
- `missing_recommended_fields`

این تصمیم باعث می‌شود محصول در شرایط واقعی‌تر مانند ازدحام اورژانس، کمبود زمان یا قطع بخشی از اطلاعات قابل استفاده‌تر باشد.

# پیش نویس گزارش نهایی پروژه

## عنوان

سیستم هوشمند پشتیبان تصمیم گیری تریاژ اورژانس

## خلاصه مدیریتی

این پروژه با هدف طراحی و پیاده سازی یک MVP مبتنی بر هوش مصنوعی برای کمک به تریاژ اورژانس انجام شده است. سامانه پیشنهادی با دریافت داده های قابل دسترسی در لحظه تریاژ، مانند علائم حیاتی، سن، روش ورود و شکایت اصلی، احتمال بحرانی بودن بیمار را تخمین می زند و یک توضیح کوتاه برای تصمیم ارائه می کند.

مسئله پروژه مستقیما با ارزش انسانی و اجتماعی گره خورده است: در شرایط ازدحام، بحران، جنگ، حادثه جمعی یا کمبود منابع، تشخیص سریع تر بیماران پرخطر می تواند به کاهش رنج مردم و استفاده بهتر از ظرفیت درمانی کشور کمک کند. این پروژه تلاش می کند نشان دهد چگونه یک تیم دانشجویی می تواند با رویکرد چابک، AI و مستندسازی شفاف، یک راه حل قابل دفاع و قابل توسعه بسازد.

## انطباق با موضوع درس

موضوع درس بر توسعه سامانه های هوشمند برای مدیریت بحران تاکید دارد. تریاژ اورژانس یکی از نقاط حیاتی مدیریت بحران سلامت است؛ زیرا در لحظه فشار، تصمیم درست درباره اولویت رسیدگی می تواند اثر مستقیم بر جان انسان ها داشته باشد.

انطباق های اصلی:

- AI-enabled system
- MVP قابل اجرا
- mobile-first و API-first
- استفاده از داده در شرایط محدود
- مستندسازی Agile
- risk register، backlog، status report و AI usage report
- توجه به ذی نفع انسانی و مسئولیت اجتماعی

## محصول ساخته شده

خروجی فعلی پروژه شامل سه بخش است:

1. مدل ML نسخه v5
2. backend با FastAPI
3. frontend mobile-first برای demo

مدل فقط از داده های قابل دفاع در لحظه تریاژ استفاده می کند. داده های بعد از تریاژ مانند آزمایش ها، تصویربرداری، داروها، disposition و تشخیص های بعدی کنار گذاشته شده اند تا مدل در سناریوی واقعی قابل دفاع باشد.

## متریک های مدل v5

| معیار | مقدار تست |
|---|---:|
| AUC | 0.8917 |
| Average Precision | 0.7981 |
| Recall | 0.9194 |
| Precision | 0.5275 |
| FPR | 0.3572 |
| Threshold | 0.3061 |

تفسیر مدیریتی: پروژه عمدا Recall را مهم تر از Precision می داند، چون از دست دادن بیمار بحرانی هزینه انسانی بالاتری نسبت به بررسی بیشتر یک بیمار غیر بحرانی دارد. با این حال FPR نسبت به نسخه قبلی گزارش شده، یعنی 0.487، کاهش محسوسی داشته است.

## قابلیت کار با داده ناقص

یکی از الزامات کاربردی پروژه این است که در محیط واقعی همیشه همه اطلاعات کامل نیستند. بنابراین API و frontend طوری طراحی شده اند که با چند آیتم اولیه نیز کار کنند. خروجی سامانه علاوه بر probability، سطح کامل بودن داده و فهرست فیلدهای پیشنهادی باقی مانده را نمایش می دهد.

این تصمیم باعث می شود محصول در سناریوهای واقعی تر مانند ازدحام، کمبود زمان یا قطعی بخشی از اطلاعات قابل استفاده تر باشد.

## معماری

- ML: مدل XGBoost انتخاب شده بر اساس validation
- Backend: FastAPI با endpointهای `/health`، `/model-info` و `/predict`
- Frontend: رابط mobile-first و فارسی برای ورود سریع اطلاعات بیمار، سناریوهای demo و نمایش سطح کامل بودن داده
- Explainability: توضیح کوتاه مبتنی بر علائم حیاتی و شکایت اصلی
- Documentation: مستندات زنده پروژه در پوشه `docs`

## رویکرد Agile

پروژه به صورت sprintهای کوتاه مدیریت شده است. به جای تلاش برای ساخت محصول کامل از ابتدا، ابتدا مدل و مستندات پایه آماده شد، سپس backend و در ادامه frontend و artefactهای ارائه تکمیل شدند.

artefactهای مدیریتی:

- backlog
- risk register
- status report
- model card
- AI usage report
- lessons learned
- poster content
- presentation outline

## استفاده از AI

AI در این پروژه برای توسعه سریع تر، تحلیل مدل، مستندسازی، طراحی معماری و آماده سازی artefactهای مدیریتی استفاده شده است. با این حال خروجی AI بدون کنترل انسانی پذیرفته نشده و معیارهای کمی مدل، تست API و مستندات برای اعتبارسنجی استفاده شده اند.

## ریسک ها و اخلاق

مهم ترین ریسک ها:

- False Negative در تشخیص بیمار بحرانی
- اتکای بیش از حد کاربر به مدل
- bias نسبت به گروه های جمعیتی
- نبود اعتبارسنجی بالینی واقعی
- محرمانگی داده در صورت استفاده عملی

اصل اخلاقی پروژه: این سامانه فقط decision-support است و جایگزین پزشک، پرستار یا پروتکل رسمی بیمارستان نیست.

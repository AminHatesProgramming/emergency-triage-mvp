# مستند معماری سیستم

## نمای کلی

سامانه «پشتیبان تصمیم گیری تریاژ اورژانس» یک محصول AI-enabled و crisis-oriented است که برای کمک به تشخیص سریع بیماران بحرانی در محیط اورژانس طراحی می شود. معماری MVP به صورت mobile-first، API-first و قابل توسعه برای حالت offline در نظر گرفته شده است.

## اهداف معماری

- کاهش احتمال از دست رفتن بیمار بحرانی
- ارائه خروجی قابل توضیح برای افزایش اعتماد کاربر بالینی
- امکان اجرای سریع MVP در محیط آموزشی/نمایشی
- جداسازی منطق مدل از رابط کاربری
- فراهم کردن مسیر توسعه برای داده واقعی بیمارستانی

## اجزای اصلی

### 1. لایه داده

منبع فعلی داده، فایل `data/raw/triage.csv` است. این داده شامل متغیرهای جمعیت شناختی، شکایت اصلی، علائم حیاتی، سوابق بالینی و سطح ESI است.

داده های اصلی مورد استفاده در MVP:

- سن و گروه سنی
- جنسیت
- روش ورود بیمار
- علائم حیاتی اولیه: ضربان قلب، فشار خون، تنفس، اکسیژن خون و دما
- شکایت اصلی بیمار با featureهای `cc_*`
- سطح تریاژ ESI به عنوان برچسب آموزشی

### 2. لایه مهندسی ویژگی

در نسخه فعلی مدل، علاوه بر ویژگی های خام، ویژگی های بالینی مشتق شده نیز ساخته شده اند:

- `shock_index`
- `map`
- `pulse_pressure`
- `hr_rr_ratio`
- `vital_severity_score`
- `is_elderly`
- `is_pediatric`

هدف این ویژگی ها این است که مدل فقط به الگوهای آماری سطحی تکیه نکند و نشانه های بالینی معنادار را نیز دریافت کند.

### 3. لایه مدل

نسخه حرفه ای تر مدل در `ml/train.py` با رویکرد ensemble و کنترل leakage ساخته می شود:

- Random Forest با `class_weight='balanced_subsample'`
- Extra Trees برای پایداری بیشتر در داده های tabular
- XGBoost با `scale_pos_weight`
- مقایسه مدل های تکی با ensemble بر اساس AUC validation
- انتخاب predictor نهایی: XGBoost در اجرای v5
- انتخاب threshold روی validation، نه test

خروجی مدل احتمال بحرانی بودن بیمار است. در ادامه، با threshold تنظیم شده برای هدف Recall، بیمار به دو دسته «بحرانی» و «غیربحرانی» تقسیم می شود.

متریک تست v5: AUC برابر 0.8917، Recall برابر 0.9194، Precision برابر 0.5275 و FPR برابر 0.3572 است.

### 4. لایه API

در MVP، backend با FastAPI طراحی شده و نسخه اولیه آن در `backend/main.py` آماده است. این API مدل v5 را lazy load می کند و برای demo کلاس قابل استفاده است.

Endpointهای پیشنهادی:

- `GET /health`: بررسی سلامت سرویس
- `POST /predict`: دریافت اطلاعات بیمار و بازگشت خروجی تریاژ
- `GET /model-info`: نمایش نسخه مدل، threshold و معیارهای ارزیابی

نمونه خروجی پیش بینی:

```json
{
  "risk_level": "critical",
  "critical_probability": 0.91,
  "recommended_action": "Immediate clinical review",
  "model_version": "v4",
  "explanation": [
    "low oxygen saturation",
    "high shock index",
    "elderly patient"
  ]
}
```

### 5. لایه رابط کاربری

رابط کاربری باید mobile-first باشد، چون کاربر هدف در سناریوی اورژانس نیاز به ورود سریع اطلاعات دارد.

ویژگی های مورد انتظار UI:

- فرم سریع علائم حیاتی
- انتخاب شکایت اصلی
- نمایش سطح تریاژ پیشنهادی
- نمایش درصد اطمینان
- نمایش توضیح کوتاه و قابل فهم
- ثبت هشدار اخلاقی: «این خروجی جایگزین تصمیم متخصص نیست.»

### 6. توضیح پذیری

برای توضیح پذیری، خروجی SHAP در فایل `shap_esi1.png` تولید شده است. در MVP، توضیح ها می توانند ابتدا به صورت rule-based و بر اساس featureهای مهم تولید شوند و در نسخه بعدی به SHAP آنلاین یا batch متصل شوند.

## تصمیمات معماری و منطق آنها

| تصمیم | دلیل |
|---|---|
| API-first | جداسازی مدل از UI و امکان تست مستقل |
| Mobile-first | تناسب با محیط سریع و پرتنش اورژانس |
| Ensemble ML | کاهش وابستگی به یک مدل و بهبود پایداری |
| Threshold مبتنی بر Recall | اولویت انسانی پروژه، از دست ندادن بیمار بحرانی است |
| Explainability | افزایش اعتماد و امکان دفاع در ارائه |
| Offline-capable roadmap | کاربرد بالقوه در بحران و قطعی زیرساخت |

## محدودیت های فعلی

- frontend هنوز کامل نشده است.
- مدل فعلی روی داده ثانویه آموزش دیده و نیازمند اعتبارسنجی با ذی نفع واقعی است.
- اجرای SHAP آنلاین ممکن است برای MVP سنگین باشد.
- فایل های مدل حجیم هستند و باید برای استقرار نهایی بهینه شوند.

## مسیر تکامل معماری

1. تکمیل API پیش بینی
2. اتصال API به مدل ذخیره شده
3. ساخت UI موبایل
4. افزودن توضیح پذیری قابل نمایش
5. تست با سناریوهای نمونه
6. دریافت بازخورد از کاربر یا متخصص حوزه سلامت
7. آماده سازی نسخه نمایشی برای ارائه نهایی

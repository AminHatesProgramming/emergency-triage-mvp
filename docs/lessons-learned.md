# درس آموخته ها

این سند به صورت زنده در طول پروژه تکمیل می شود و برای خودارزیابی، ارائه وضعیت و تحویل نهایی استفاده خواهد شد.

## درس های فنی

| موضوع | مشاهده | نتیجه |
|---|---|---|
| Calibration | استفاده از `CalibratedClassifierCV` در نسخه قبلی باعث افت عملی و FPR بالا شد | calibration باید با validation جدا و هدف پروژه هماهنگ باشد |
| Class imbalance | بیماران بحرانی نسبت به کل داده کمتر هستند | معیار اصلی نباید فقط accuracy باشد |
| Threshold | threshold پیش فرض 0.5 با هدف انسانی پروژه سازگار نیست | threshold باید بر اساس Recall هدف تنظیم شود |
| Feature engineering | featureهای خام کافی نیستند | featureهای بالینی مشتق شده باعث معنی دارتر شدن مدل می شوند |
| Explainability | تمرکز SHAP روی age/gender می تواند نشانه ریسک bias باشد | باید featureهای بالینی تقویت و bias پایش شود |
| Test leakage | انتخاب threshold روی test گزارش را غیرقابل دفاع می کند | در v5 threshold روی validation انتخاب شد و test فقط برای گزارش نهایی استفاده شد |
| Resource usage | خواندن کل دیتاست با 972 ستون باعث خطای حافظه شد | در v5 فقط ستون های لازم با `usecols` خوانده می شوند |

## درس های مدیریت پروژه

- مستندات باید همزمان با توسعه تکمیل شوند، نه فقط در پایان ترم.
- برای پروژه AI محور، معیارهای موفقیت باید از ابتدا عددی و قابل دفاع باشند.
- backlog باید به خروجی های قابل نمایش در هر sprint شکسته شود.
- پوستر پروژه یک خروجی یک بار مصرف نیست؛ یک سند زنده برای نمایش پیشرفت است.
- تصمیم های فنی باید با ارزش اجتماعی و محدودیت زمانی پروژه هماهنگ باشند.

## درس های کار با AI

- AI برای سرعت دادن به تولید پیش نویس و تحلیل مفید است، اما صحت سنجی آن ضروری است.
- در پروژه سلامت، ادبیات مستندات باید محافظه کارانه و دقیق باشد.
- استفاده از AI باید ثبت شود تا در ارزیابی درس شفاف باشد.
- خروجی مدل و خروجی ابزار زبانی هر دو نیاز به ارزیابی انسانی دارند.

# یادداشت تکمیل نسخه موبایل و PWA

## هدف

در فاز نهایی، MVP از یک فرم وب ساده به یک تجربه mobile-first و قابل ارائه تبدیل شد. تمرکز این فاز روی سه نیاز استاد و سناریوی واقعی اورژانس بود:

- کارکرد قابل دفاع با داده ناقص
- نمایش شفاف احتمال مدل، پرچم ایمنی و اقدام بعدی
- آماده بودن برای demo موبایلی و ویدئوی ۱۰ دقیقه‌ای

## تغییرات محصول

| بخش | خروجی |
|---|---|
| Frontend موبایل | action bar پایین صفحه، تب‌های ورودی/خروجی/بازخورد، فرم فشرده، کارت نتیجه قابل اسکن |
| PWA | `manifest.webmanifest`، `sw.js`، آیکن‌های 192/512 و cache شدن app shell بعد از بارگذاری اول |
| سناریوهای دمو | بحرانی، متوسط، کم‌داده و قلبی |
| Explainability | عوامل موثر، safety flags، next best actions و فیلدهای پیشنهادی باقی‌مانده |
| جداسازی محصول از گزارش | KPI، sprint، backlog و knowledge base از UI کاربر حذف شدند و فقط در مستندات رسمی باقی ماندند |
| Demo polish | خلاصه قابل کپی/چاپ برای هر کیس، راهنمای نصب موبایل و فرم بازخورد نسخه آزمایشی |
| Feedback export | ثبت server-side بازخورد و خروجی CSV از `/feedback/export` برای ضمیمه گزارش، بدون نمایش دکمه مدیریتی در UI کاربر |

## Safety-first Hybrid Inference

مدل v7 پایه تصمیم است و `model_probability` احتمال خام مدل را نشان می‌دهد. برای کاربرد اورژانس، یک لایه guard شفاف اضافه شد که red flagهای ساده و قابل دفاع را جداگانه برمی‌گرداند:

- اکسیژن خون پایین
- فشار سیستولیک کمتر از ۹۰
- نرخ تنفس یا ضربان قلب به‌شدت غیرطبیعی
- دمای بدن در محدوده بحرانی
- درد قفسه سینه همراه با سابقه قلبی پرخطر

این guard جایگزین مدل یا متخصص نیست؛ فقط برای کاهش خطر از دست رفتن بیمار بحرانی در شرایط داده ناقص استفاده می‌شود.

| جداسازی لینک‌های مدیریتی از UI محصول | Pass |

## نکته ارائه

در ویدئو، بهتر است ابتدا سناریوی کم‌داده یا قلبی نمایش داده شود. این کار نشان می‌دهد سامانه فقط دنبال یک دقت عددی نیست، بلکه برای محیط واقعی اورژانس، داده ناقص، شفافیت و مسئولیت اخلاقی طراحی شده است.

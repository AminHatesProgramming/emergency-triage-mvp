## 1. خلاصه نهایی پروژه برای پوستر

### عنوان پروژه
تریاژ هوشمند اورژانس

### زیرعنوان
سامانه پشتیبان تصمیم‌گیری برای شناسایی سریع‌تر بیماران بحرانی با کمک هوش مصنوعی

### بیان مسئله
در اورژانس‌های شلوغ، فشار کاری و کمبود زمان می‌تواند باعث تأخیر در شناسایی بیماران پرخطر شود. این تأخیر برای بیماران بحرانی پیامد انسانی جدی دارد.

### راه‌حل ارائه‌شده
یک MVP فارسی و mobile-first طراحی شد که با داده‌های در دسترس در زمان تریاژ، احتمال بحرانی بودن بیمار را تخمین می‌زند، هشدارهای ایمنی را نمایش می‌دهد و اقدام بعدی قابل بازبینی پیشنهاد می‌کند.

### ارزش‌آفرینی محصول
- کاهش احتمال از دست رفتن بیمار بحرانی در شرایط ازدحام
- کمک به تصمیم‌گیری سریع‌تر و مستندتر در تریاژ
- پشتیبانی از ورودی ناقص و چند داده کلیدی
- ارائه خروجی قابل توضیح برای کاربر غیرمتخصص فنی

### ارزش اجتماعی و ملی
این پروژه با هدف کمک به تاب‌آوری نظام سلامت کشور، استفاده بهتر از ظرفیت محدود اورژانس، و کاهش ریسک خطای انسانی در موقعیت‌های پرفشار طراحی شده است.

### یادداشت اخلاقی
این سامانه جایگزین پزشک، پرستار یا پروتکل رسمی بیمارستان نیست. خروجی آن فقط یک ابزار پشتیبان تصمیم است و باید زیر نظر متخصص تفسیر شود.

## 2. اطلاعات نهایی مدل

| شاخص | مقدار نهایی |
|---|---:|
| نسخه مدل نهایی | v7 |
| مدل عملیاتی انتخاب‌شده | XGBoost v7 balanced / safety-first hybrid |
| AUC تست | 0.9041 |
| Recall بیماران بحرانی | 0.9246 |
| Precision | 0.5447 |
| FPR | 0.3352 |
| Threshold نهایی | 0.3017 |
| Average Precision | 0.8202 |
| تعداد ردیف تست | 111,606 |
| تعریف بیمار بحرانی | ESI 1 یا ESI 2 |

### داده‌های استفاده‌شده در مدل
مدل فقط از داده‌هایی استفاده می‌کند که در زمان تریاژ یا در شرح حال/پرونده کوتاه اولیه قابل دسترسی هستند: علائم حیاتی اولیه، شکایت اصلی، سن، جنسیت، روش ورود، سابقه مراجعه/بستری/جراحی و سوابق بالینی شناخته‌شده.

### پشتیبانی از ورودی ناقص
سامانه با ورودی ناقص نیز ارزیابی اولیه ارائه می‌دهد. UI و API برای سناریوی sparse تست شده‌اند و خروجی شامل `data_completeness` و فیلدهای پیشنهادی باقی‌مانده است.

### دلیل قابل دفاع بودن مدل
- حذف داده‌های پس از تریاژ مانند آزمایش، دارو، تصویربرداری، disposition و تشخیص‌های بعدی
- انتخاب threshold روی validation، نه test
- استفاده از test فقط برای گزارش نهایی
- تمرکز روی Recall به دلیل هزینه انسانی بالای False Negative
- ثبت محدودیت‌ها و نیاز به اعتبارسنجی بالینی واقعی

## 5. مدیریت پروژه و Agile

### رویکرد مدیریت پروژه
رویکرد پروژه Agile / Scrum سبک‌وزن است: هر Sprint یک خروجی قابل تحویل دارد و تغییرات مدل، API، UI، مستندات و بازخورد کاربر به صورت تدریجی مدیریت شده‌اند.

### Sprintها و خروجی هر Sprint

| Sprint | بازه | خروجی قابل تحویل |
|---|---|---|
| Sprint 0 | 2026-06-01 | تعریف مسئله، scope، stakeholder register |
| Sprint 1 | 2026-06-02 تا 2026-06-03 | baseline مدل، کنترل leakage، feature set اولیه |
| Sprint 2 | 2026-06-04 تا 2026-06-05 | FastAPI، endpointهای `/health` و `/predict`، سناریوی sparse |

### دستاوردهای اخیر

- اضافه شدن نصب موبایل با راهنمای fallback

### برنامه مرحله بعد

- مستندات پروژه باید همزمان با توسعه تکمیل شوند.
- محصول کاربرمحور باید از گزارش مدیریتی جدا بماند.
- در پروژه سلامت، محافظه‌کاری اخلاقی بخشی از کیفیت فنی است.

### محدودیت صادقانه
این MVP با داده ثانویه آموزش دیده و برای استفاده واقعی نیازمند اعتبارسنجی بالینی، بررسی fairness و تأیید نهادی است.
